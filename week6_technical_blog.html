04/18/14 - Object-Oriented Programming vs. Functional Programming

Object-Oriented Programming (OOP)

Ruby is an example of an OO language. In OOP, also called imperative programming, we rely on classes which consist mainly of a collection of method definitions. The purpose of a class is to create new objects which are called instances of a class. The instances of a class have their individual state and behaviour. OOP mimics the real world in which the objects' state can be modified by different behaviour. Lets say the state of an instance of a class Person is as follows:

Example:

@mood = 'good'
@weight = weight

An instance has its possible behaviours defined in the instance methods of its class, in our case class Person.

Example:

#instance method
def eat
@weight + 5
unless @weight > 60 # kg
@mood
end
@mood = "bad"
end

As we see, depending on the instance's behaviour (#eat), it can change its state: mood and weight.


Functional programming (FP)

In contrast to OOP, FP relies on functions rather than objects and their state. FP emphasises mathematical functions that produce results that depend only on their inputs. i.e. we can say: In FP, same input means always same output.

FP is implemented in languages like Haskell but also the good old in Excel is based on it. FP uses expressions, i.e. something that has a value. This can be an integer (4), a string ('hello') or a boolean (true). An expression is not stored in a variable like in OOP, but given to a function directly. In FP, a function (f) now works with these expressions (x) that are explicitly given to it but:

In FP, the function (f) does something with the input (x), and that is exactly what we define in the function (f), but the state of the input (x) will never be changed while the function is been executed.

In other words: the state of the expression x in f(x) will never change as expressions are not mutable in FP.
